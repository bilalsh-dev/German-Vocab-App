# Feature 03 — Internationalization (EN / DE)

We're localizing the interface in English and German. This covers **UI chrome only** — card content (German vocabulary, English prompts) is data, never i18n keys. Depends on Feature 02 (shell) for the language switcher slot.

## Library

Use `next-intl`. Install and configure it for the App Router. Confirm with the owner before installing if a different i18n library is preferred (see open question in `progress-tracker.md`).

## Locale Files

- `messages/en.json` and `messages/de.json`, mirrored key structure.
- Organize keys by surface (e.g. `nav.*`, `footer.*`, `decks.*`, `study.*`, `editor.*`, `common.*`).
- Replace every placeholder string from Feature 02 with a key.

## Language Switching

- The language switcher in the top nav toggles EN / DE.
- Persist the chosen language in the `settings` store (Feature 04 provides it; until then, persist to a **cookie** (`wortbox.locale`) and migrate later — see decision note below).
- Apply the language on load so the choice survives restarts.

> **Persistence decision (2026-06-18):** The interim mechanism is a cookie, not `localStorage`. A cookie is readable on the server, so `next-intl` resolves the correct locale during SSR and the right language renders on first paint — `localStorage` would force a client-only resolution causing a hydration mismatch / flash for a non-default locale. The switcher writes the cookie via a `next-intl`-style Server Action (`setUserLocale`), which triggers a re-render so the UI updates immediately. This migrates to the `settings` store in Feature 04.

## Constraints

- No hardcoded UI strings anywhere — all interface text resolves through `next-intl`.
- Do not put vocabulary or example sentences in locale files.

### Check when done
- Switching language updates all visible UI chrome immediately.
- Language choice persists across reload.
- A search for hardcoded user-facing strings in shell/screen components comes back clean.
- `pnpm test`, `pnpm run lint`, and `pnpm run build` pass.
